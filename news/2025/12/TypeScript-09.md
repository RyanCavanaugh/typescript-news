### [Issue microsoft/TypeScript#54925](https://github.com/microsoft/TypeScript/issues/54925) (Open, `Suggestion`, `Feature Update`)

**Numeric Range Types \(Feature Update\)**

*Introduce numeric and character range types (0..15, 0...1, 'a'..'d') to represent bounded values in type definitions.*

 * [52 weeks ago](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-2524231128) **fredrare** suggested implementing as a first-class type and shared a NumberRange type gist for small ranges
 * [50 weeks ago](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-2551900259) **HansBrende** offered a fourth option for the resulting type as a union of two ranges
 * [29 weeks ago](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-2889633151) **KrystilizeNevaDies** asked why the type could not be left as-is and only the new range type supported
 * [today](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-3633325194) **mkantor** explained that Exclude<0..32767, 5> would yield the full 0..32767 range rather than exclude 5 and that treating numeric ranges as distributive would produce a 32766-member union causing out-of-memory issues
 * [today](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-3633483207) **snarbles2** asked whether a numeric range should be treated as an optimized union and explained how Exclude<0..32767, 5> distributes and can be represented as ranges
 * [today](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-3633527696) **mkantor** agreed with previous points and explained that he was answering why the type couldn't remain as-is, noting he might use different terminology for its member count
 * [later](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-3636306426) **minecrawler** proposed a Range type requiring bounds-checking instead of union types, provided examples including exclusions and subranges, and raised questions about special number bounds, BigInt, floating-point handling, subtype relationships, and language precedents
 * [later](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-3636576684) **KrystilizeNevaDies** asked why `0..32767` being a subtype of `5` mattered and proposed expanding conditional types to simplify logic
 * [later](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-3636600953) **shaedrich** suggested differentiating integer and floating-point ranges using `0..100` and `0...100` syntax, noting Ruby's similar inclusive/exclusive range notation
 * [later](https://github.com/microsoft/TypeScript/issues/54925#issuecomment-3636617328) **KrystilizeNevaDies** said "@shaedrich I feel like if a float implementation is possible, an int implementation is pretty low priority since we should be able to model all int operations as types."
